# 13. Discovery-решения PACK.IT

Этот документ фиксирует ответы брейншторма и превращает их в решения, требования и открытые вопросы.

## Блок A. Продукт и границы системы

### Вопрос A1

Какой самый частый реальный сценарий использования PACK.IT от первого контакта с клиентом до закрытия проекта?

### Ответ / исходное описание

После встречи с клиентом определяется состав проекта, дата и промежуток времени проведения мероприятия.

Пользователь с нужными правами — менеджер, директор или технический директор — создаёт проект, указывает клиента, выбирая его из существующей базы или добавляя нового, заполняет информацию о месте проведения, адресе и времени проведения мероприятия.

Отдельно нужно продумать, как и на каком этапе указывать время монтажа и демонтажа.

Если нужен выезд технического директора на площадку, пользователь прожимает отдельный чекпоинт и назначает технического директора проекта. После назначения технический директор получает уведомление о необходимости посетить площадку и провести исследование по чеклисту с документированием и фотографированием всех технических аспектов.

Следующим шагом пользователь выбирает наполнение проекта: звук, свет, сцена, фермы, backline, LED и другие нужные разделы. После этого заполняется смета.

Если нужен 3D-визуализатор, пользователь включает его и в нём собирает сцену, ферменные конструкции, LED-экран и другие визуальные элементы.

Все объекты, построенные в 3D-конструкторе, по умолчанию должны попадать в итоговую смету и техническую комплектацию, если пользователь явно не указал, что объект нужен только для визуализации.

Для 3D-объектов нужна механика выбора режима:

```text
1. Учитывать в смете / BOM
2. Только показать на схеме / визуализации
```

Это важно, потому что в 3D-сцене пользователь может накидать сцену, фермы, LED, колонки, световые приборы и другие объекты как для расчёта, так и просто для схемы площадки.

После заполнения сметы указывается транспорт: куда, сколько, каким транспортом и по какой цене.

Следующим шагом назначаются пользователи на проект.

### Зафиксированные решения

#### Decision A1.1 — Главный сценарий начинается после встречи с клиентом

PACK.IT должен поддерживать сценарий, где проект создаётся после предварительного обсуждения или встречи с клиентом, когда уже известны общий состав проекта, дата и временной интервал мероприятия.

#### Decision A1.2 — Проект создаёт пользователь с правом на создание проекта

Создавать проект может не только менеджер, но любой пользователь, у которого есть соответствующее право: менеджер, директор, технический директор или другая роль.

#### Decision A1.3 — Клиент выбирается из базы или создаётся на месте

При создании проекта пользователь должен иметь возможность выбрать существующего клиента или добавить нового, не покидая сценарий создания проекта.

#### Decision A1.4 — Выезд технического директора оформляется чекпоинтом

В проекте должен быть отдельный чекпоинт: нужен ли выезд технического директора на площадку.

Если чекпоинт включён, пользователь назначает технического директора, а назначенный пользователь получает уведомление и задачу на обследование площадки.

#### Decision A1.5 — Обследование площадки должно иметь чеклист и фотофиксацию

Технический директор должен иметь инструмент для проведения площадочного исследования: чеклист, заметки, фотографии и фиксация технических аспектов.

#### Decision A1.6 — Разделы проекта выбираются перед заполнением сметы

Пользователь должен выбрать наполнение проекта: звук, свет, сцена, фермы, LED, backline, услуги и другие разделы. После выбора разделов пользователь заполняет смету.

#### Decision A1.7 — 3D-конструктор является опциональным инструментом проекта

3D-конструктор включается только если он нужен для проекта. Он используется не только для расчёта, но и для итоговой визуализации и схемы площадки.

#### Decision A1.8 — 3D-объект может быть расчётным или визуальным

Каждый объект в 3D-сцене должен иметь признак, участвует ли он в смете/BOM или используется только для визуализации.

Пример:

```text
Колонка поставлена в 3D-сцену:
- вариант 1: добавить в смету и складскую потребность;
- вариант 2: оставить только на схеме для визуализации.
```

#### Decision A1.9 — Транспорт идёт после наполнения сметы

После заполнения основных разделов сметы пользователь указывает транспорт: направление, количество, тип транспорта и стоимость.

#### Decision A1.10 — Назначение пользователей идёт после формирования проекта

После сметы, 3D и транспорта пользователь назначает участников проекта: технических специалистов, водителей, складских участников и другие роли.

### Требования

#### Requirement A1.1 — Создание проекта

Система должна позволять создать проект с указанием клиента, площадки, адреса, даты и временного интервала мероприятия.

#### Requirement A1.2 — Быстрое создание клиента

Если клиента нет в базе, пользователь должен иметь возможность создать клиента прямо из формы проекта.

#### Requirement A1.3 — Чекпоинт выезда технического директора

В проекте должен быть чекпоинт `Нужен выезд технического директора на площадку`.

При включении чекпоинта система должна предложить назначить технического директора.

#### Requirement A1.4 — Уведомление технического директора

После назначения технический директор должен получить уведомление о необходимости посетить площадку.

#### Requirement A1.5 — Чеклист площадки

Система должна поддерживать чеклист обследования площадки с возможностью прикреплять фотографии и комментарии.

#### Requirement A1.6 — Выбор разделов проекта

В проекте должен быть шаг выбора разделов: звук, свет, сцена, фермы, LED, backline, услуги, транспорт и прочие.

#### Requirement A1.7 — Опциональный запуск 3D-конструктора

В проекте должен быть чекпоинт или действие для включения 3D-конструктора.

#### Requirement A1.8 — Режим 3D-объекта

У каждого объекта 3D-сцены должен быть режим участия:

```text
quote_bom — участвует в смете и BOM
visual_only — используется только для схемы/визуализации
```

#### Requirement A1.9 — Передача 3D-объектов в смету

Если 3D-объект имеет режим `quote_bom`, он должен попадать в итоговую смету, техническую комплектацию и складскую потребность.

#### Requirement A1.10 — Визуальные объекты не должны попадать в склад

Если 3D-объект имеет режим `visual_only`, он не должен попадать в итоговую смету, BOM, складскую потребность и резерв.

#### Requirement A1.11 — Транспортный шаг

В проекте должен быть транспортный шаг с указанием маршрута, количества, типа транспорта и стоимости.

#### Requirement A1.12 — Назначение команды проекта

После формирования проекта пользователь должен иметь возможность назначить пользователей и роли на проект.

### Открытые вопросы

#### Open Question A1.1 — Монтаж и демонтаж

На каком этапе пользователь должен указывать время монтажа и демонтажа: сразу при создании проекта, после обследования площадки, после подтверждения сметы или отдельным шагом календаря?

#### Open Question A1.2 — Очередность технического директора

Должен ли выезд технического директора происходить до заполнения сметы, после предварительной сметы или как параллельная задача?

#### Open Question A1.3 — Автоматическое включение 3D

Должна ли система сама предлагать включить 3D, если пользователь выбрал разделы сцена, фермы или LED?

#### Open Question A1.4 — Режим по умолчанию для 3D-объектов

Какой режим должен быть по умолчанию для объектов, добавленных в 3D:

```text
quote_bom
```

или

```text
visual_only
```

#### Open Question A1.5 — Назначение команды

Команда назначается после транспорта или часть ролей должна назначаться раньше, например технический директор на этапе обследования площадки?

### Вопрос A2

Какие типы проектов PACK.IT должен поддерживать в первой рабочей версии?

### Ответ / исходное описание

Первая рабочая версия должна поддерживать все типы проектов. Если ограничить систему только узкими типами проектов, приложение получится слишком условным и даже на первых этапах будет применимо только к узкому набору задач.

### Зафиксированные решения

#### Decision A2.1 — Проект не ограничивается жёстким типом

PACK.IT не должен запрещать создание проекта только потому, что он не подходит под заранее заданный тип.

Проект может быть любым: концерт, корпоратив, свадьба, конференция, аренда оборудования, работа команды, транспортная задача, субаренда, сцена, фермы, LED, звук, свет, backline или произвольная комбинация разделов.

#### Decision A2.2 — Тип проекта является шаблоном, а не ограничением

Тип проекта может использоваться как стартовая заготовка, которая предлагает набор разделов и настроек, но не ограничивает пользователя.

Примеры:

```text
Полный концерт → звук + свет + сцена + фермы + LED + команда + транспорт
Аренда звука → звук + транспорт
Сцена и фермы → сцена + фермы + транспорт + команда
Пустой проект → пользователь сам выбирает разделы
```

#### Decision A2.3 — Универсальный проект важнее набора узких калькуляторов

Первая версия должна строиться вокруг универсального проекта и универсального конструктора разделов. Глубина автоматизации может отличаться по разделам, но пользователь не должен упираться в невозможность создать нестандартный проект.

### Требования

#### Requirement A2.1 — Универсальное создание проекта

Система должна позволять создать проект без выбора жёсткого типа.

#### Requirement A2.2 — Проектные шаблоны

Система может предлагать шаблоны проектов, но пользователь должен иметь возможность менять состав разделов после выбора шаблона.

#### Requirement A2.3 — Произвольный набор разделов

Пользователь должен иметь возможность собрать проект из произвольного набора разделов.

#### Requirement A2.4 — Custom-раздел

В первой рабочей версии должен быть механизм произвольного раздела или произвольных строк, чтобы покрывать нестандартные задачи до появления специализированной автоматизации.

### Вопрос A3

Что PACK.IT точно не должен пытаться делать на старте?

### Ответ / исходное описание

Функции вроде бухгалтерии, полноценной кассы, сложных внешних интеграций, маркетинга и других крупных дополнительных направлений не должны входить в первую версию. Но под такие возможности стоит заложить фундамент, чтобы при необходимости их можно было безболезненно подключить позже.

### Зафиксированные решения

#### Decision A3.1 — Не делать всё сразу

Первая версия не должна превращаться в комбайн, который одновременно пытается быть сметчиком, складом, бухгалтерией, кассой, маркетингом, договорным сервисом, мессенджером и ERP.

#### Decision A3.2 — Будущие модули должны быть предусмотрены архитектурно

Даже если модуль не входит в первую версию, архитектура должна позволять подключить его позже без болезненной переделки ядра.

#### Decision A3.3 — Core и Future Modules должны быть разделены

В документации и кодовой архитектуре нужно разделять:

```text
Core MVP — то, без чего первая версия не имеет смысла
Future Module — то, что может быть подключено позже
Integration Adapter — внешний сервис или интеграция
```

### Не входит в стартовую версию, но может быть заложено архитектурно

- полноценная бухгалтерия;
- касса и фискализация;
- онлайн-оплата;
- интеграции с бухгалтерскими системами;
- интеграции с внешними календарями;
- Telegram-боты;
- email-рассылки и маркетинг;
- полноценная CRM-воронка продаж;
- автоматическая генерация договоров;
- электронный документооборот;
- сложная аналитика и BI;
- мультикомпании / SaaS white-label, если это будет мешать первой версии;
- маркетинговые страницы и публичные кабинеты клиентов;
- сложный мобильный 3D-редактор, если он затормозит создание основного продукта.

### Требования

#### Requirement A3.1 — Модульная архитектура

Система должна проектироваться так, чтобы будущие модули подключались через понятные интерфейсы, а не через переписывание ядра.

#### Requirement A3.2 — Событийная модель

В системе желательно заложить событийную модель для будущих интеграций:

```text
project.created
project.approved
project.cancelled
quote.sent
inventory.reserved
inventory.issued
document.generated
user.assigned
```

#### Requirement A3.3 — Integration layer

Даже если внешние интеграции не реализуются в первой версии, архитектура должна предусматривать слой адаптеров для будущих подключений.

#### Requirement A3.4 — Не блокировать MVP будущими модулями

Будущие возможности не должны усложнять первую версию настолько, чтобы задержать запуск основного проектно-сметного сценария.
