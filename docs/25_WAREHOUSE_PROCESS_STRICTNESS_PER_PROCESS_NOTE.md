# 25 Warehouse — Process Strictness Per Process Note

## 10.108. Где хранить настройку строгости складских процессов

Решение: настройку строгости склада лучше делать гибко по отдельным процессам, а не одним глобальным режимом на всю компанию.

Основной принцип:
- администратор должен иметь возможность включать нужный уровень строгости в нужный момент;
- разные складские процессы могут требовать разной строгости;
- компания может начинать с гибкого режима и постепенно ужесточать отдельные участки, когда появляется необходимость.

Настройки строгости должны быть доступны отдельно для процессов:
- инвентаризация;
- выдача;
- возврат;
- списания;
- потери / утери;
- ремонт / сервис;
- изменение места хранения;
- переносы и отмены плановых проверок;
- проблемные строки.

Пример:
- инвентаризация работает в строгом режиме: обязательные причины, история переносов, эскалации;
- выдача работает в гибком режиме: минимум обязательных полей;
- списания работают строго всегда, потому что требуют подтверждения директора/админа;
- возврат может быть средним: обычный возврат простой, проблемные строки требуют комментария.

Правило:
- не делать один общий переключатель `гибкий/строгий` на весь склад как единственный механизм;
- предусмотреть административные настройки по отдельным процессам;
- по умолчанию для MVP использовать гибкие настройки, кроме критичных действий вроде списаний и потерь;
- админ может включать более строгий режим постепенно, когда компания до этого дорастает.

Архитектурный смысл:
- складской модуль должен адаптироваться под реальную зрелость компании;
- маленькому прокату не нужна лишняя бюрократия во всех процессах сразу;
- большой склад может формализовать именно те участки, где есть риск ошибок, потерь или спорных ситуаций;
- это лучше масштабируется, чем один глобальный режим строгости.

MVP-подход:
- заложить модель настроек `warehouse_process_rules` или аналогичную;
- хранить режим/флаги по процессам;
- начать с простых флагов: обязательный комментарий, эскалация, лимит переносов, строгий дедлайн, расширенная история;
- интерфейс настроек доступен администратору/директору.
