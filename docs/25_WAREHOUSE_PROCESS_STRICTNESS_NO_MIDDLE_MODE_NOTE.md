# 25 Warehouse — Process Strictness No Middle Mode Note

## 10.109. Промежуточный режим строгости

Решение: промежуточный режим строгости между гибким и строгим не нужен.

Основная модель:
- гибкий режим;
- строгий режим;
- отдельные флаги и правила по конкретным складским процессам.

Причина:
- режим `нормальный` между гибким и строгим может усложнить настройки без большой практической пользы;
- гибкость лучше давать не третьим общим режимом, а точечной настройкой правил внутри каждого процесса;
- администратору должно быть понятно, какие конкретные требования включены: обязательный комментарий, дедлайн, эскалация, лимит переносов, расширенная история и т.д.

Правило:
- не вводить триаду `гибкий / нормальный / строгий` как основную модель;
- оставить два базовых уровня: гибкий и строгий;
- детализацию реализовывать через набор флагов процесса;
- разные процессы могут иметь разные настройки без введения промежуточного режима.

Пример:
- инвентаризация: строгий режим + обязательная причина переноса;
- выдача: гибкий режим без лишних обязательных полей;
- возврат: гибкий режим, но проблемные строки требуют комментарий;
- списания: строгие правила и подтверждение директора/админа.

Архитектурный смысл:
- настройки остаются понятными;
- система не плодит лишние абстрактные режимы;
- администратор управляет конкретными правилами, а не пытается угадать, что означает `нормальный` режим.

MVP-подход:
- хранить базовый режим процесса как `flexible` или `strict`;
- дополнительные требования хранить отдельными флагами;
- не добавлять режим `normal` на старте.
