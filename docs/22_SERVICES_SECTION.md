# 22. Услуги

## Решение

`Услуги` — отдельный раздел сметы и проекта.

Услуги — это работы и операции, а не люди. Люди и роли живут в разделе `Команда`.

Раздел должен быть расширяемым: стартовый список услуг является базовым, но справочник можно дополнять без изменения кода.

## Клиентский вывод

В клиентской смете услуги могут показываться отдельными строками или группироваться по решению менеджера.

Пример:

```text
Услуги
Монтаж — цена Y
Настройка — цена Y
Демонтаж — цена Y
```

Клиент видит понятные названия услуг и итоговую стоимость. Внутренние формулы, себестоимость и маржа клиенту по умолчанию не показываются.

## Стартовые типы услуг

На старте в справочник услуг входят:

- монтаж;
- настройка;
- демонтаж;
- погрузка / разгрузка;
- доставка как услуга, если она не заведена в транспорте;
- выезд специалиста;
- подготовка / предпродакшн;
- другое.

Этот список не финальный. Если появляются повторяющиеся операции, их нужно добавлять в справочник услуг.

## Расширяемость справочника

Справочник услуг должен позволять добавлять новые типы без изменения кода.

Для услуги желательно хранить:

- название;
- клиентское название;
- категория / группа;
- единица расчёта;
- ставка по умолчанию;
- внутренняя себестоимость по умолчанию, если нужна;
- цена клиенту по умолчанию;
- показывать клиенту: да / нет;
- комментарий.

## Типы расчёта

Услуга может считаться разными способами:

- фиксированная цена;
- за час;
- за смену;
- за единицу оборудования / комплекта;
- за рейс / операцию;
- вручную.

Система может подставлять значения из справочника, но менеджер должен иметь возможность изменить цену и комментарий в конкретном проекте.

## Внутренняя стоимость, цена клиенту и маржа

Строка услуги должна хранить внутреннюю экономику и клиентскую цену отдельно.

Для каждой строки услуги хранятся:

- тип услуги;
- количество;
- тип расчёта: фикс / час / смена / операция / вручную;
- внутренняя себестоимость;
- цена клиенту;
- маржа;
- комментарий;
- показывать клиенту: да / нет.

Формула маржи:

```text
маржа = цена клиенту − внутренняя себестоимость
```

Клиент видит только клиентскую цену. Внутренняя себестоимость и маржа используются для управленческого учёта и не показываются клиенту по умолчанию.

Система может подставлять базовые значения из справочника, но менеджер может менять внутреннюю себестоимость, клиентскую цену и тип расчёта в конкретном проекте.

## Услуги в итоговой марже проекта

Услуги участвуют в общей экономике проекта как полноценный финансовый блок.

Маржа проекта должна учитывать:

- маржу оборудования;
- маржу услуг;
- маржу команды;
- маржу транспорта;
- маржу субаренды;
- расходники, если у них есть клиентская цена и себестоимость.

Это позволяет видеть реальную управленческую картину проекта, а не только маржу по оборудованию.

## Скидка и услуги

По умолчанию услуги не входят в базу скидки.

Базовое правило скидки:

- скидка применяется к оборудованию / аренде;
- услуги исключаются из скидки по умолчанию;
- команда исключается из скидки по умолчанию;
- транспорт исключается из скидки по умолчанию;
- субаренда исключается из скидки по умолчанию;
- расходники исключаются из скидки по умолчанию;
- ручные позиции исключаются из скидки по умолчанию.

Менеджер может явно включить конкретную строку или раздел в скидку, если это нужно для проекта.

## Автоматический расчёт услуг от оборудования

Некоторые услуги могут рассчитываться от количества оборудования или объёма работ.

Примеры:

- монтаж LED — от количества кабинетов;
- монтаж сцены — от количества модулей;
- монтаж ферм — от метров, узлов или сложности конструкции;
- погрузка — от объёма, количества мест или условных единиц.

Система может предлагать автоматический расчёт, но менеджер может заменить его вручную.

Автоматический расчёт не должен молча блокировать ручное решение менеджера. Если менеджер изменил расчёт вручную, система должна сохранять это как ручное значение для конкретного проекта.

## Видимость услуг в клиентской смете

Для клиентской сметы у раздела `Услуги` есть три режима видимости:

- `детально` — услуги выводятся отдельными строками;
- `общим блоком` — одна строка, например `Услуги`;
- `скрыть из клиентской сметы` — услуги остаются только во внутренней экономике проекта.

В любом режиме внутри проекта сохраняется полная детализация по типам услуг, расчётам, себестоимости, клиентской цене и марже.

## Связь услуги с командой

Связь услуги с командой должна быть мягкой.

Пример:

- услуга: `Монтаж сцены`;
- возможные роли команды: `техник сцены`, `монтажник`, `грузчик`.

Система может подсказывать, какие роли обычно нужны для выбранной услуги, но не должна автоматически создавать людей в команде без решения менеджера.

Услуга продаётся клиенту отдельно, а команда назначается отдельно.

Такой подход сохраняет гибкость: услугу можно добавить в смету без немедленного назначения исполнителей, а команду можно собрать позже с учётом занятости, ставок и фактической логистики проекта.

## Автоматические подсказки услуг

Система должна помогать неопытным менеджерам и подсказывать типовые услуги на основе добавленных разделов и оборудования.

Примеры подсказок:

- добавили сцену → подсказать `монтаж сцены`, `демонтаж сцены`;
- добавили фермы → подсказать `монтаж ферм`, `такелаж / подвес`, `демонтаж ферм`;
- добавили LED → подсказать `монтаж LED`, `настройка LED`, `демонтаж LED`;
- добавили звук → подсказать `монтаж звука`, `настройка звука`, `демонтаж звука`.

Система только предлагает услуги, а менеджер сам решает, что добавить в смету.

Автоподсказки не должны молча менять смету, цену или состав проекта без подтверждения менеджера.

## Автоматические подсказки команды по услугам

Система может подсказывать роли команды на основе выбранных услуг.

Пример:

- услуга: `монтаж LED`;
- подсказанные роли: `LED-инженер`, `монтажник`, `техник LED`.

Система не назначает конкретных людей автоматически. Менеджер сам выбирает исполнителей с учётом занятости, ставок, опыта, доступа и логистики проекта.

Подсказки команды нужны как помощь менеджеру, особенно если он не знает полный состав ролей для конкретной услуги.

## Разграничение с командой

`Команда` — это роли и назначенные люди.

`Услуги` — это работы и операции, которые продаются клиенту или учитываются во внутренней экономике проекта.

Пример:

- `Техник сцены` — команда;
- `Монтаж сцены` — услуга.

Эти сущности связаны, но не заменяют друг друга.

## Разграничение с транспортом

Если доставка оформляется как отдельный транспортный блок, она живёт в разделе `Транспорт`.

Если доставка нужна как простая услуга без детального расчёта транспорта, она может быть строкой в `Услугах`.

Важно не дублировать одну и ту же доставку одновременно в `Услугах` и `Транспорте`.
