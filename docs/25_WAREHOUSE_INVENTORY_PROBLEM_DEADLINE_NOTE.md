# 25.x. Дедлайн разбора проблемной строки инвентаризации

Этот документ дополняет `docs/25_WAREHOUSE_INVENTORY_PROBLEM_STATUSES_NOTE.md` и фиксирует решение 10.87.

## Решение

Проблемная строка инвентаризации может иметь необязательный дедлайн разбора.

Дедлайн не должен быть обязательным для MVP и маленьких прокатов, но его стоит заложить в фундамент архитектуры.

## Логика

По умолчанию дедлайн не обязателен.

Компания или администратор могут включить настройки:

- не использовать дедлайны;
- использовать дедлайн только для критичных проблем;
- требовать дедлайн для всех проблемных строк.

## Если дедлайн указан

Если дедлайн указан, система может показывать предупреждение ответственному, когда срок приближается или уже прошёл.

Для критичных просроченных проблем можно дополнительно уведомлять администратора или директора.

## Почему так

В маленькой компании обязательные дедлайны могут быть лишней бюрократией.

В большой компании дедлайны помогают не оставлять проблемные строки без разбора месяцами.

## MVP

Для MVP достаточно:

1. необязательное поле дедлайна;
2. отображение дедлайна в проблемной строке;
3. фильтр по просроченным строкам;
4. возможность включить строгий режим позже через настройки.
