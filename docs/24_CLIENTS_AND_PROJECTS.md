# 24. Клиенты и проекты

## Решение

`Клиенты и проекты` — базовый стартовый блок CRM.

Клиент и проект создаются до состава сметы, потому что все дальнейшие разделы сметы, логистики, команды, склада, документов и финансов привязываются к конкретному проекту.

## Клиент

Клиент нужен как основная сущность, к которой привязываются проекты, контакты, история, документы, счета, сметы и коммуникации.

На старте клиент может быть выбран из базы или создан в процессе создания проекта.

## Тип клиента

На старте у клиента есть тип:

- физическое лицо;
- юридическое лицо / организация.

Тип клиента нужен для будущей логики документов, реквизитов, договоров, счетов и отображения данных в проекте.

Для MVP тип клиента не должен усложнять создание проекта: достаточно простого выбора типа при создании или редактировании клиента.

## Физическое лицо

Для клиента типа `физическое лицо` на старте нужны поля:

- имя;
- фамилия;
- телефон;
- email;
- комментарий.

Этих полей достаточно для быстрого создания клиента, связи с проектом, коммуникаций и базового документооборота на старте.

## Юридическое лицо / организация

Для клиента типа `юридическое лицо / организация` карточку нужно разделить на чистую основную часть и отдельный блок реквизитов.

В основной карточке организации на старте нужны поля:

- название организации;
- контактное лицо;
- телефон;
- email;
- комментарий.

ИНН, КПП и остальные реквизиты юридического лица не должны засорять основную карточку.

Реквизиты нужно сгруппировать в отдельном окошке / блоке карточки клиента, чтобы интерфейс оставался чистым.

## Реквизиты юридического лица

Реквизиты юридического лица должны находиться в отдельном блоке / окошке карточки клиента.

На старте в блоке реквизитов нужны поля:

- ИНН;
- КПП;
- ОГРН;
- юридический адрес;
- фактический адрес;
- банковские реквизиты;
- комментарий по реквизитам.

Также нужна возможность загрузить файл с реквизитами.

Файл с реквизитами должен быть доступен для просмотра из карточки клиента.

Это позволит быстро прикреплять готовые реквизиты клиента, не засоряя основную карточку организации большим количеством полей.

## Проект

Проект — рабочая единица CRM.

К проекту привязываются:

- состав сметы;
- оборудование;
- услуги;
- команда;
- транспорт;
- субаренда;
- расходники;
- питание 220/380;
- инфраструктура площадки;
- площадка и зал;
- документы;
- задачи;
- финансы;
- комментарии;
- история изменений.

## Поля проекта при создании

Для MVP при создании проекта нужны поля:

- название проекта;
- клиент;
- дата мероприятия;
- площадка / адрес;
- зал / пространство площадки, если есть;
- ответственный менеджер;
- комментарий.

Этого достаточно, чтобы быстро создать проект, связать его с клиентом и перейти к составу сметы.

## Быстрый проект / быстрая аренда со склада

Для MVP нужен быстрый сценарий создания проекта без полноценного оформления клиента, чтобы быстро отдать в аренду несколько позиций со склада.

Такой сценарий не должен быть совсем без данных: всё равно нужно фиксировать, кто забрал оборудование и на какой период.

Минимальные данные быстрого сценария:

- кто забрал оборудование;
- контакт / телефон, если есть;
- что выдали со склада;
- дата и время выдачи;
- дата и время возврата;
- цена клиенту;
- внутренняя себестоимость;
- залог;
- статус оплаты: оплачено / не оплачено;
- комментарий.

При быстром сценарии система должна:

- зарезервировать позиции на складе на указанный период;
- внести занятость оборудования в календарь;
- сохранить связь выдачи с проектом / быстрым проектом;
- сразу зафиксировать простую финансовую часть аренды;
- позволить позже дополнить данные клиента и проекта, если сделка вырастет в полноценный проект.

Финансовые поля быстрого сценария нужны для контроля денег, залога, себестоимости и маржи даже в небольших складских арендах.

Это нужно для быстрых складских аренд без перегруза полным проектным flow.

## Залог в быстрой аренде

Для залога в быстрой аренде недостаточно одного текстового поля.

На старте нужны поля:

- сумма залога;
- получен / не получен;
- возвращён / не возвращён;
- комментарий по залогу.

Это нужно, чтобы контролировать не только факт залога, но и его возврат после завершения аренды.

Залог не должен смешиваться с выручкой проекта и маржей. Он фиксируется отдельно как временно удерживаемая сумма.

## Акт выдачи для быстрой аренды

Для быстрой аренды со склада обязательно нужен простой акт / документ выдачи.

Документ должен формироваться в PDF и включать:

- перечень выданного оборудования;
- кто забрал оборудование;
- контакт / телефон, если указан;
- дату и время выдачи;
- дату и время возврата;
- цену / сумму аренды;
- сумму залога;
- статус оплаты;
- комментарий;
- место для подписей.

Акт выдачи нужен для контроля передачи оборудования, срока возврата, финансовых условий и ответственности сторон.

Для MVP достаточно простого PDF без сложного документооборота и электронных подписей.

## Возврат оборудования по быстрой аренде

Для возврата оборудования по быстрой аренде на старте не нужен отдельный акт возврата.

Достаточно отметить возврат в карточке быстрой аренды / акте выдачи.

При отметке возврата желательно фиксировать:

- дату и время фактического возврата;
- кто принял оборудование;
- комментарий по состоянию оборудования;
- отметку по возврату залога, если применимо.

После возврата система должна освободить складской резерв и обновить календарь занятости оборудования.

## Статусы быстрой аренды

Для быстрой аренды нужен отдельный простой статус.

На старте достаточно статусов:

- выдано;
- возвращено;
- просрочено;
- отменено.

Статус нужен для складского контроля, календаря занятости оборудования и быстрого понимания, какие выдачи ещё не закрыты.

## Доступ к быстрой аренде

Быстрая аренда должна быть складским инструментом, а не отдельным полноценным проектным модулем.

При этом доступ к инструменту должен быть не только у сотрудников склада.

На старте доступ нужен:

- складу;
- менеджерам;
- аккредитованным работникам / доверенным сотрудникам.

Такой доступ нужен, чтобы быстро выдавать небольшие складские аренды, но не открывать этот сценарий всем пользователям системы.

Права доступа должны ограничивать, кто может создать быструю аренду, выдать оборудование, отметить возврат и закрыть аренду.

## Аккредитация для складских операций

Аккредитованный работник — это не отдельная роль в системе.

Нужно использовать признак / галочку у пользователя:

```text
аккредитован для складских операций
```

Этот признак даёт доступ к разрешённым складским действиям, включая быструю аренду, если это разрешено политикой доступа.

Такой подход лучше, чем отдельная роль, потому что пользователь может оставаться менеджером, техником или другим сотрудником, но дополнительно иметь доверенный доступ к складским операциям.

Права по признаку должны быть ограниченными и не должны автоматически давать полный доступ к складу, финансам или админке.

## Фото быстрой аренды

В быстрой аренде должна быть возможность прикреплять фото при выдаче и возврате оборудования.

Это нужно для разрешения спорных ситуаций по состоянию оборудования.

Примеры фото:

- фото оборудования при выдаче;
- фото комплектации / серийных номеров, если нужно;
- фото оборудования после возврата;
- фото повреждений;
- фото спорных мест или отсутствующих элементов.

Фото должны храниться в карточке быстрой аренды и быть связаны с конкретной выдачей.

На старте достаточно возможности прикрепить фото и комментарий к ним без сложного фотоотчёта.

## Статусы проекта

На старте у проекта есть простой список статусов:

- черновик;
- в работе;
- согласован;
- выполнен;
- отменён.

Этого достаточно для MVP, чтобы понимать состояние проекта без сложного workflow.

## Площадки

Площадка должна быть отдельной сущностью / базой, а не только текстовым полем в проекте.

Разные клиенты могут проводить мероприятия на одной и той же площадке, поэтому площадку нужно выбирать из базы прошлых площадок или создавать новую.

У площадки могут быть разные залы / пространства для проведения мероприятий.

Пример:

```text
Площадка: Дом культуры
Зал: Большой зал
Зал: Малый зал
Зал: Фойе
```

В проекте выбирается площадка и, если нужно, конкретный зал / пространство.

Это позволит копить историю проектов по площадке, быстрее заполнять адреса и в будущем хранить технические особенности площадки.

## Карточка площадки

Для карточки площадки на старте нужны поля:

- название площадки;
- адрес;
- контакт площадки;
- телефон;
- комментарий;
- список залов / пространств.

Также в карточке площадки нужно хранить местных специалистов и полезные контакты.

Примеры локальных контактов:

- администратор площадки;
- электрик;
- технический директор площадки;
- ответственный за зал;
- охрана / пропускной режим;
- другой местный специалист.

Для каждого такого контакта желательно хранить:

- роль / должность;
- имя;
- телефон;
- комментарий.

Это нужно, чтобы менеджер или техдир быстро находил нужного человека на площадке при подготовке проекта и на монтаже.

## Карточка зала / пространства площадки

Для зала / пространства внутри площадки на старте нужны поля:

- название зала / пространства;
- этаж / расположение;
- вместимость, если известно;
- габариты, если известно;
- высота потолка, если известно;
- размер штатной сцены, если она есть;
- доступное подключение к электричеству;
- максимальная доступная мощность на площадке / в зале;
- комментарий.

Для доступного подключения к электричеству нужно фиксировать, что реально есть на площадке:

- Schuko по стенам;
- CEE 32;
- CEE 63;
- хвост / прямое подключение к автомату;
- другое.

Эти данные нужны для подготовки питания 220/380, проверки нагрузки, подбора генератора, технического листа и предварительной оценки ограничений площадки.

## Фото и файлы площадки / зала

Для площадки и зала нужно хранить фото и файлы, но не в основной карточке.

Фото и файлы должны быть вынесены в отдельную вкладку или отдельное окно, чтобы не перегружать основной интерфейс.

Примеры файлов и фото:

- фото зала;
- схема площадки;
- план эвакуации;
- технический райдер площадки;
- фото электрощитов / вводов;
- фото штатной сцены;
- прочие файлы.

Эти материалы нужны для подготовки проекта, технического планирования, питания, монтажа и коммуникации с площадкой.

## История проектов по площадке

В карточке площадки нужно показывать последние 4–5 проектов, которые проходили на этой площадке.

Полная история не должна перегружать карточку на старте.

В компактном списке последних проектов желательно показывать:

- дату;
- клиента;
- название проекта;
- зал / пространство;
- комментарий.

Это поможет быстро вспомнить прошлый опыт работы на площадке и найти похожий проект.

## История проектов по клиенту

В карточке клиента нужно показывать последние 4–5 проектов этого клиента.

Полная история не должна перегружать карточку на старте.

В компактном списке последних проектов желательно показывать:

- дату;
- название проекта;
- площадку;
- статус;
- сумму.

Это поможет быстро оценить историю отношений с клиентом и перейти к похожему прошлому проекту.

## Создание проекта

Перед переходом к составу сметы нужно выбрать или создать клиента и создать проект.

После создания проекта пользователь попадает на экран состава сметы, где выбирает разделы проекта и наполняет их.

## Роль клиента в других разделах

Выбранный клиент используется в других разделах проекта.

Примеры:

- в самовывозе достаточно клиента, выбранного при создании проекта;
- в документах клиент подставляется в КП, договоры и счета;
- в истории клиента сохраняются прошлые проекты;
- в коммуникациях проектные сообщения можно связать с клиентом.

## MVP-ограничение

На старте не нужно превращать блок клиентов в тяжёлую enterprise CRM.

Главная задача MVP:

- быстро создать клиента;
- быстро создать проект;
- перейти к составу сметы;
- сохранить связь проекта со всеми дальнейшими разделами.
