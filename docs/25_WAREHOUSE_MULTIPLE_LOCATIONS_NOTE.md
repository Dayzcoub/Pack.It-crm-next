# 25.x. Несколько складских локаций

Этот документ дополняет `docs/25_WAREHOUSE_LAYOUT_CONSTRUCTOR_NOTE.md` и фиксирует решение 10.53.

## Решение

В архитектуре склада нужно поддерживать несколько складских локаций.

Это нужно компаниям, у которых два или более склада, помещений или постоянных мест хранения.

При этом для компании с одним складом интерфейс не должен усложняться.

## Логика

Складская локация — верхний уровень хранения.

Внутри локации могут быть:

- зоны;
- стеллажи;
- полки / уровни;
- ячейки;
- комментарии;
- 2D-план, если включён визуальный склад.

## Примеры локаций

Система должна позволять создать несколько локаций, если это нужно компании.

Каждая локация может иметь свой план, структуру хранения и список позиций.

## Поведение для маленькой компании

Если у компании один склад, система может работать в простом режиме:

- одна локация по умолчанию;
- без обязательного выбора склада каждый раз;
- место хранения внутри карточки позиции заполняется по простой структуре.

## Поведение для нескольких складов

Если локаций несколько, пользователь должен иметь возможность:

- выбрать складскую локацию;
- видеть остатки по конкретной локации;
- видеть общий остаток по всем локациям;
- запускать инвентаризацию по конкретной локации;
- указывать место хранения внутри выбранной локации.

## MVP

Для MVP достаточно заложить поддержку нескольких локаций в модели данных и интерфейсе настроек склада.

Полноценные межскладские перемещения можно не делать на старте, если ранее принято решение не усложнять внутренние перемещения.

Главное — чтобы будущий визуальный план склада мог существовать отдельно для каждой складской локации.
