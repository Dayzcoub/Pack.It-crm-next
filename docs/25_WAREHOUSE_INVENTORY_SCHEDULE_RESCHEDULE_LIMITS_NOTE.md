# 25 Warehouse — Inventory Schedule Reschedule Limits Note

## 10.100. Ограничение количества переносов плановой инвентаризации

Решение: количество переносов плановой инвентаризации не должно быть жёстко ограничено одинаково для всех компаний. Механизм нужно сделать гибким и настраиваемым.

Логика по типу компании:
- для маленьких прокатов и небольших команд переносы должны оставаться максимально гибкими;
- для большой компании со складскими работниками можно включать более строгий контроль переносов;
- история всех переносов сохраняется всегда, независимо от режима.

Возможные режимы настройки:
- без ограничения переносов;
- предупреждение после заданного количества переносов;
- обязательная причина переноса всегда;
- эскалация директору/администратору после заданного количества переносов;
- запрет дальнейшего переноса без повышенных прав после заданного лимита.

Архитектурный смысл:
- PACK.IT должен подходить и маленькому прокату, где процессы часто гибкие и ручные, и большой компании, где складской контроль должен быть формализован;
- ограничение переносов — это не базовое жёсткое правило, а административная настройка;
- факт каждого переноса сохраняется в истории: старая дата, новая дата, пользователь, время изменения и комментарий причины.

MVP-подход:
- по умолчанию не ограничивать количество переносов;
- обязательно хранить историю переносов;
- предусмотреть настройку строгого режима для будущего расширения или для компаний с формальным складским процессом.
