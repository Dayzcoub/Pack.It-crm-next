# 14. Жизненный цикл проекта

## 1. Статус документа

Этот документ фиксирует решения по жизненному циклу проекта PACK.IT.

## 2. Первое решение по завершению проекта

На первых этапах проект можно помечать завершённым вручную.

Система не должна на старте жёстко блокировать завершение проекта из-за незаполненных автоматических условий.

## 3. Почему ручное завершение важно на старте

В реальной работе проект может считаться завершённым по-разному:

- мероприятие прошло;
- оборудование вернулось;
- склад закрыл возврат;
- команда отработала;
- документы сформированы;
- клиенту всё отправлено;
- внутренние расчёты закрыты;
- долг или оплата отмечены вручную;
- проект больше не требует активных действий.

Если на старте сделать слишком строгую автоматическую систему закрытия, она может мешать работе и блокировать реальные сценарии.

## 4. Требование первой версии

В первой рабочей версии у пользователя с нужными правами должна быть возможность вручную перевести проект в статус:

```text
completed
```

или пользовательское отображение:

```text
Завершён
```

## 5. Рекомендуемый чеклист перед завершением

Перед переводом проекта в завершённый статус система может показывать мягкий чеклист:

- мероприятие прошло;
- оборудование выдано и возвращено;
- складские резервы закрыты;
- команда назначена и отработала;
- транспорт выполнен;
- документы сформированы;
- финансовые отметки внесены вручную;
- комментарии по проекту заполнены.

На старте этот чеклист должен быть рекомендательным, а не блокирующим.

## 6. Права доступа

Завершать проект должен только пользователь с соответствующим правом.

Пример права:

```text
project.complete
```

Возможные роли:

- владелец пространства;
- директор;
- менеджер проекта;
- технический директор;
- другой пользователь, которому явно выдано право.

## 7. История изменений

При ручном завершении проекта система должна записывать событие в историю:

```text
project.completed
```

Событие должно содержать:

- кто завершил проект;
- когда завершил проект;
- предыдущий статус;
- новый статус;
- комментарий, если пользователь его указал.

## 8. Будущая автоматизация

Позже можно добавить автоматические подсказки и проверки:

- есть незакрытые складские резервы;
- есть невозвращённое оборудование;
- есть незавершённые задачи;
- не сформированы документы;
- не назначена команда;
- есть открытый дефицит;
- есть неоплаченные суммы или ручные финансовые отметки.

Но эти проверки не должны быть обязательным условием первой версии.

## 9. Открытые вопросы

### Open Question LIFE.1 — Нужно ли поле комментария при завершении

Нужно решить, должен ли пользователь обязательно писать комментарий при завершении проекта или комментарий должен быть опциональным.

### Open Question LIFE.2 — Кто может переоткрыть проект

Нужно решить, кто может вернуть завершённый проект обратно в работу.

### Open Question LIFE.3 — Архив

Нужно решить, отличается ли статус `completed` от `archived`, или архив — это отдельное действие после завершения.
