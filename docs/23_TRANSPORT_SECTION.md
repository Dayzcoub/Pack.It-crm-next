# 23. Транспорт

## Решение

`Транспорт` — отдельный раздел сметы и проекта.

Транспорт нужен не только для клиентской цены, но и для внутренней логистики, себестоимости, маржи, планирования рейсов и контроля исполнителей.

## Клиентский вывод

В клиентской смете транспорт показывается отдельным разделом или несколькими строками по типам / единицам транспорта.

Пример:

```text
Транспорт
Грузовой транспорт — цена Y
Легковой транспорт — цена Y
```

Клиент видит понятное название транспортной строки и итоговую стоимость. Внутренняя себестоимость, маржа и логистические детали клиенту по умолчанию не показываются.

## Стартовые типы транспорта

На старте в справочник типов транспорта входят:

- грузовой;
- легковой;
- прицеп;
- микроавтобус;
- газель / малый грузовой;
- фура / большой грузовой;
- манипулятор;
- такси / каршеринг;
- курьер;
- другое.

Список должен быть расширяемым без изменения кода.

## Ручной транспорт

Менеджер может добавить транспорт вручную, не выбирая его из справочника.

Поля ручного транспорта:

- название;
- тип транспорта;
- маршрут;
- количество / рейсы;
- цена клиенту;
- внутренняя себестоимость;
- комментарий.

Ручной транспорт не обязан становиться позицией справочника.

Кнопку `сохранить в справочник` прямо из сметы не делаем. Часто повторяющиеся транспортные позиции нужно добавлять в справочник отдельно через нормальный административный процесс.

## Поля транспортной строки

Для каждой транспортной строки желательно хранить:

- тип транспорта;
- исполнитель;
- маршрут;
- количество / рейсы;
- дата подачи;
- время подачи;
- дата возврата / обратного рейса, если есть;
- время возврата;
- тип расчёта себестоимости;
- цена клиенту;
- внутренняя себестоимость;
- маржа;
- комментарий;
- комментарий по таймингу.

## Исполнитель транспорта

Исполнитель может быть:

- наш транспорт;
- наёмный транспорт;
- субарендатор;
- самовывоз / клиент забирает сам;
- другое.

Исполнитель нужен для внутренней логистики, себестоимости и ответственности.

Сценарий `самовывоз / клиент забирает сам` особенно важен, когда компания отдаёт своё оборудование в субаренду и клиент или партнёр сам забирает оборудование.

## Самовывоз

При самовывозе отдельные поля `кто забирает`, `телефон`, `контакт самовывоза` на старте не нужны.

Достаточно клиента, который уже выбран в начале создания проекта.

Если требуется дополнительная информация по самовывозу, её можно указать в комментарии транспортной строки или в комментарии проекта.

## Ответственный за транспорт

Отдельное поле ответственного за транспорт на старте не нужно.

Транспортом занимается тот же человек, который отвечает за проект.

Это упрощает MVP и не плодит лишние роли, если в реальном процессе транспорт контролирует один ответственный менеджер проекта.

## Маршрут

Для MVP основной формат маршрута простой:

```text
откуда → куда
```

При необходимости можно добавить промежуточные точки.

На старте не нужно превращать раздел `Транспорт` в полноценную TMS-систему со сложной оптимизацией маршрутов, складами, трекингом и детальным диспетчерским контуром.

Главная задача маршрута на первом этапе — зафиксировать понятную логистику проекта, себестоимость и ответственность.

## Время транспорта

Для транспорта на старте нужны простые поля времени:

- дата подачи;
- время подачи;
- дата возврата / обратного рейса, если есть;
- время возврата;
- комментарий по таймингу.

Эти поля нужны для внутренней логистики, планирования рейсов, задач и контроля ответственных.

Клиенту эти детали по умолчанию не показываются. В клиентской смете остаётся итоговая транспортная строка и цена.

## Статусы транспорта

На старте отдельные статусы транспорта не добавляем.

Статусы вроде `запланирован`, `подтверждён`, `в пути`, `выполнен`, `отменён` для MVP не нужны и усложняют процесс без явной необходимости.

Контроль транспорта на первом этапе ведётся через поля маршрута, времени, исполнителя и комментарии.

## Типы расчёта себестоимости

На старте поддерживаются:

- фикс за рейс / транспорт;
- ставка × часы;
- ставка × километры;
- вручную.

`Ставка × километры` нужна для такси, курьера, наёмного транспорта и ситуаций, где себестоимость зависит от расстояния.

`Вручную` нужна для нестандартных случаев, когда формула не подходит или менеджер вводит согласованную сумму сам.

Клиенту показывается итоговая сумма строки, а не внутренняя формула расчёта.

## Маржа транспорта

Маржа транспорта считается отдельно.

```text
маржа транспорта = цена клиенту − транспортная себестоимость
```

Маржа транспорта входит в общую экономику проекта.

## Скидка и транспорт

По умолчанию транспорт не входит в базу скидки.

Если менеджеру нужно сделать скидку на транспорт, он может явно включить конкретную транспортную строку в скидку.

Внутренняя себестоимость и маржа транспорта при этом сохраняются отдельно и используются для управленческого учёта.

## Связь с субарендой

Доставка или забор субарендных позиций может входить в транспортную себестоимость проекта.

Если доставку / забор делает субарендатор, стоимость доставки фиксируется отдельно и плюсуется в транспортные расходы проекта.

Важно не дублировать одну и ту же доставку одновременно в `Транспорте`, `Услугах` и строках субаренды.

## Разграничение с услугами

Если доставка оформляется как детальный транспортный блок, она живёт в разделе `Транспорт`.

Если доставка нужна как простая услуга без детального расчёта транспорта, она может быть строкой в `Услугах`.

Одна и та же доставка не должна дублироваться в двух разделах.
