# 25 Warehouse — Process Strictness Level Note

## 10.107. Настройка уровня строгости складских процессов

Решение: для складского блока PACK.IT нужно предусмотреть настройку уровня строгости процессов.

Смысл настройки:
- один и тот же складской модуль должен подходить маленькому прокату, где важна скорость и гибкость;
- и большой компании, где нужны формальные правила, обязательные комментарии, эскалации и контрольные статусы;
- строгие правила не должны быть одинаково навязаны всем компаниям по умолчанию.

Базовые режимы:

### Гибкий режим

Подходит для маленьких прокатов и небольших команд.

Характеристики:
- минимум обязательных полей;
- меньше блокирующих проверок;
- больше ручных управленческих решений;
- переносы и отмены плановых проверок доступны директору/админу без избыточной бюрократии;
- система сохраняет историю, но не перегружает рабочий интерфейс контрольными деталями.

### Строгий режим

Подходит для больших компаний и складов с отдельными складскими работниками.

Характеристики:
- больше обязательных комментариев;
- можно включать лимиты и предупреждения по переносам;
- можно включать эскалации ответственному, администратору или директору;
- можно строже контролировать просрочки, проблемные строки и отмены;
- история изменений может отображаться подробнее.

Правило:
- уровень строгости должен быть административной настройкой компании/рабочего пространства;
- по умолчанию для MVP логичен гибкий режим;
- строгие правила должны включаться осознанно директором/администратором;
- отдельные настройки можно расширять постепенно, не ломая базовый складской workflow.

Архитектурный смысл:
- PACK.IT должен масштабироваться от маленькой команды до большой компании;
- один и тот же модуль не должен быть слишком слабым для крупного склада и слишком бюрократичным для небольшого проката;
- настройка строгости позволяет сохранить баланс между скоростью работы и управляемостью.

MVP-подход:
- заложить концепцию `warehouse_process_strictness`;
- стартовое значение: гибкий режим;
- строгий режим можно реализовывать постепенно через отдельные флаги: обязательные комментарии, лимиты переносов, эскалации, дедлайны и расширенное отображение истории.
