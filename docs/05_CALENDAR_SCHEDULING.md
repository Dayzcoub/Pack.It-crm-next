# 05. Календарь и планирование

## 1. Статус модуля

Календарь должен входить в стартовую версию PACK.IT.

Это не future module и не внешняя интеграция на потом. Это базовый модуль проектной работы.

## 2. Почему календарь нужен на старте

Проект в PACK.IT связан не только с коммерческой сметой, но и с реальными датами и действиями:

- дата и время мероприятия;
- период проведения мероприятия;
- монтаж;
- демонтаж;
- выезд технического директора на площадку;
- дедлайны подготовки;
- назначение команды;
- подготовка склада;
- выдача оборудования;
- возврат оборудования;
- транспорт;
- контрольные точки проекта.

Без календаря невозможно нормально управлять проектом, занятостью людей, складскими резервами и подготовкой мероприятия.

## 3. Главное решение

Календарь является частью ядра PACK.IT.

```text
Project / Quote
→ Schedule
→ Calendar Events
→ Tasks / Checkpoints
→ Team / Inventory / Transport
```

Календарь не должен быть отдельной игрушкой, оторванной от проекта. Все календарные события должны быть связаны с проектами, задачами, пользователями и складскими действиями.

## 4. Что календарь должен покрывать в первой версии

### 4.1. Событие проекта

Основное мероприятие:

- дата начала;
- время начала;
- дата окончания;
- время окончания;
- площадка;
- клиент;
- проект;
- статус.

### 4.2. Монтаж

Монтаж должен быть отдельным календарным событием или временным блоком проекта.

Нужно определить:

- дата монтажа;
- время начала монтажа;
- время окончания монтажа;
- ответственные;
- команда;
- транспорт;
- комментарии.

### 4.3. Демонтаж

Демонтаж должен быть отдельным календарным событием или временным блоком проекта.

Нужно определить:

- дата демонтажа;
- время начала демонтажа;
- время окончания демонтажа;
- ответственные;
- команда;
- транспорт;
- комментарии.

### 4.4. Выезд технического директора

Если в проекте включён чекпоинт выезда технического директора, система должна создать календарную задачу или событие для назначенного технического директора.

Событие должно быть связано с чеклистом площадки.

### 4.5. Складская подготовка

Календарь должен учитывать подготовительные складские действия:

- дата подготовки комплекта;
- дата выдачи;
- дата возврата;
- внутренний дедлайн сборки;
- связь с резервом.

### 4.6. Команда

Назначенные пользователи должны видеть свои проектные события:

- монтаж;
- мероприятие;
- демонтаж;
- выезд на площадку;
- отдельные задачи;
- дедлайны.

## 5. Представления календаря

Минимально нужны:

- календарь по проектам;
- календарь по пользователю;
- календарь по команде;
- календарь по складу / выдачам;
- список ближайших событий.

Возможные представления:

```text
День
Неделя
Месяц
Список
```

Для первой версии можно начать с недели, месяца и списка ближайших событий.

## 6. Связь с проектом

В карточке проекта должны отображаться ключевые даты:

- мероприятие;
- монтаж;
- демонтаж;
- выезд технического директора;
- складская подготовка;
- выдача;
- возврат.

Из календаря пользователь должен иметь возможность открыть связанный проект.

Из проекта пользователь должен иметь возможность открыть календарный блок проекта.

## 7. Связь с уведомлениями

Календарь должен быть источником уведомлений.

Примеры уведомлений:

- назначен выезд технического директора;
- завтра монтаж;
- сегодня выдача со склада;
- скоро демонтаж;
- пользователь назначен на проект;
- изменилось время мероприятия;
- изменилось время монтажа;
- проект подтверждён и требует подготовки склада.

## 8. Внешние интеграции

Внешние интеграции с Google Calendar, Apple Calendar, Outlook или Telegram можно не делать в первой версии.

Но архитектура должна предусматривать будущий export/sync adapter.

```text
Internal Calendar Event
→ Calendar Adapter
→ Google / Apple / Outlook / ICS / Telegram
```

## 9. Предварительная модель календарного события

```text
CalendarEvent
├── id
├── projectId
├── type
├── title
├── description
├── startAt
├── endAt
├── timezone
├── location
├── assignedUserIds
├── relatedChecklistId
├── relatedInventoryActionId
├── status
├── visibility
└── history
```

Типы событий:

```text
event
setup
teardown
site_survey
warehouse_prepare
warehouse_issue
warehouse_return
transport
team_task
internal_deadline
custom
```

## 10. Открытые вопросы

### Open Question CAL.1 — Где указывать монтаж и демонтаж

Нужно решить, указывается ли монтаж/демонтаж сразу при создании проекта или отдельным шагом после базовой информации о мероприятии.

### Open Question CAL.2 — Календарь склада

Нужно определить, насколько глубоко календарь склада должен быть реализован в первой версии: только даты выдачи/возврата или полноценная загрузка склада по дням.

### Open Question CAL.3 — Занятость пользователей

Нужно определить, должен ли календарь сразу проверять пересечения занятости пользователей или только показывать события без блокировки.

### Open Question CAL.4 — Внешний экспорт

Нужно определить, нужен ли экспорт ICS в первой версии, даже если полноценные внешние интеграции откладываются.
