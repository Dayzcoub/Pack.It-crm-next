# 01. Верхнеуровневая архитектура PACK.IT

## 1. Назначение системы

PACK.IT — приложение для управления техническими проектами мероприятий: от первичной сметы до комплектации, склада, документов и закрытия проекта.

Система должна поддерживать разные сценарии:

- простая смета без 3D;
- проект с техническими разделами;
- проект со складской проверкой;
- проект с 3D-сценой;
- проект с командой и транспортом;
- проект с документами для клиента и внутренней команды;
- проект с резервами, дефицитом и субарендой.

## 2. Центральная архитектурная формула

```text
Project / Quote
→ Sections
→ Optional 3D Scene
→ BOM
→ Inventory / Reservations
→ Documents
→ Status / History / CRM
```

## 3. Основные модули

### 3.1. Project / Quote Core

Главный модуль приложения.

Отвечает за:

- создание проекта;
- клиента;
- площадку;
- даты;
- статусы;
- разделы сметы;
- коммерческие суммы;
- внутреннюю техническую комплектацию;
- историю изменений;
- связь с документами, складом и 3D-сценой.

### 3.2. Sections Engine

Модуль технических разделов проекта.

Разделы могут включать:

- сцену;
- фермы;
- LED;
- звук;
- свет;
- коммутацию;
- backline;
- услуги;
- команду;
- транспорт;
- прочие позиции.

Разделы должны работать как без 3D, так и с 3D.

### 3.3. Optional 3D Scene Module

Подключаемый модуль, который открывается из проекта только при необходимости.

Отвечает за:

- 3D-сцену проекта;
- визуальное построение сцены, ферм, LED и оборудования;
- SceneModel;
- связь 3D-объектов с BOM;
- схемы для документов;
- визуализацию для клиента и технической команды.

3D-модуль не должен блокировать создание обычной сметы.

### 3.4. BOM Engine

Модуль формирования комплектации.

Отвечает за:

- нормализацию строк комплектации;
- объединение строк из разных разделов;
- расчёт количества;
- вес;
- мощность;
- стоимость;
- источник строки;
- связь со складскими позициями;
- дефициты;
- субаренду.

BOM может формироваться из ручных разделов или из 3D-сцены.

### 3.5. Inventory Module

Складской модуль.

Отвечает за:

- карточки оборудования;
- остатки;
- доступность;
- резервы;
- выдачу;
- возврат;
- списания;
- перемещения;
- дефициты;
- субаренду;
- складские документы.

Склад не управляет проектом напрямую. Проект создаёт потребность, склад её обслуживает.

### 3.6. Documents Module

Модуль документов и PDF.

Отвечает за:

- коммерческое предложение;
- техническую сводку;
- складской лист;
- монтажный лист;
- внутренний расчёт;
- схемы;
- экспорт PDF;
- шаблоны документов;
- фирменный стиль.

### 3.7. Admin Console

Системная админка приложения.

Отвечает за:

- пользователей;
- роли;
- права;
- ключи доступа;
- приглашения;
- настройки компании;
- логотипы;
- шаблоны PDF;
- справочники;
- системные настройки;
- аудит;
- управление модулями.

### 3.8. Auth / Access Control

Модуль авторизации и прав доступа.

Должен поддерживать:

- вход пользователя;
- роли;
- права на разделы;
- права на действия;
- ограниченный доступ приглашённых специалистов;
- временные и постоянные ключи;
- журнал действий;
- будущую многофакторную авторизацию.

### 3.9. Sync / Offline Module

Модуль локальной работы и синхронизации.

Должен поддерживать:

- локальные черновики;
- автосохранение;
- работу при плохом интернете;
- очередь отложенных изменений;
- безопасную синхронизацию;
- защиту от потери проекта.

## 4. Принцип модульности

Каждый модуль должен иметь понятный контракт.

Запрещено строить приложение так, чтобы UI напрямую управлял складом, документами, резервами и расчётами без доменного слоя.

Правильная схема:

```text
UI
→ Application Service
→ Domain Logic
→ Data Adapter
→ Storage / Backend
```

Неправильная схема:

```text
Button click
→ direct mutation everywhere
→ PDF / warehouse / quote / UI side effects
```

## 5. Главное ограничение

3D-сцена не должна быть единственным источником проекта.

Проект может быть создан и завершён без 3D-сцены.

Но если 3D-сцена используется, она должна отдавать результат в общий проектный контур:

```text
3D Scene
→ Scene Objects
→ BOM
→ Project Sections
→ Inventory
→ Documents
```

## 6. Предварительные пакеты будущего приложения

Возможная структура будущей кодовой базы:

```text
packages/
  core-project/
  core-sections/
  core-scene/
  core-stage/
  core-truss/
  core-led/
  core-bom/
  core-inventory/
  core-documents/
  core-admin/
  core-auth/
  core-sync/
  ui-shell/
  ui-projects/
  ui-scene-editor/
  ui-inventory/
  ui-admin/
  shared-ui/
  shared-types/
```

Эта структура не является финальной, но отражает главный принцип: расчёты, бизнес-логика, интерфейс, склад и документы не должны быть смешаны в одном монолите.
