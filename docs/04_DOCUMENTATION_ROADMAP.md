# 04. План документации PACK.IT

## 1. Цель

Перед началом активной разработки нужно подготовить такую документацию, чтобы приложение не росло хаотично.

Документация должна ответить на вопросы:

- что является ядром продукта;
- какие модули входят в систему;
- какие роли существуют;
- как устроен проект;
- как работает склад;
- когда нужен 3D-конструктор;
- как формируются документы;
- как защищаются данные;
- как приложение будет развиваться по этапам.

## 2. Уже созданные документы

- `00_PRODUCT_PRINCIPLES.md` — продуктовые принципы.
- `01_SYSTEM_ARCHITECTURE.md` — верхнеуровневая архитектура.
- `02_PROJECT_QUOTE_DOMAIN.md` — модель проекта/сметы.
- `03_ADMIN_ROLES_ACCESS.md` — системная админка, роли и доступы.

## 4. Правило именования документов

Документы нумеруются по смысловому порядку:

```text
00_...
01_...
02_...
```

Номер не означает версию. Версии продукта должны идти отдельно.

## 5. Правило изменения документов

Если решение меняется, документ нужно обновлять, а не оставлять старый текст как равноправный вариант.

Если нужно сохранить спорный вариант, его можно вынести в раздел:

```text
Open Questions
```

или

```text
Rejected Alternatives
```
