# 15. Карточка проекта и роль-зависимые представления

## 1. Статус документа

Этот документ фиксирует подход к карточке проекта PACK.IT.

## 2. Главное решение

Карточка проекта не должна быть одинаковой для всех пользователей.

То, что пользователь видит сразу после открытия проекта, зависит от:

- роли пользователя;
- его прав доступа;
- участия в конкретном проекте;
- стадии проекта;
- того, какие модули включены в проекте;
- контекста открытия: desktop, mobile, площадка, склад, офис.

## 3. Почему это важно

Один и тот же проект нужен разным людям для разных задач.

Менеджеру важны клиент, сумма, статус, PDF, согласование и состав сметы.

Техническому директору важны площадка, чеклист, технические разделы, схемы, команда, монтаж, демонтаж, риски, вес, рабочая мощность и пусковая мощность.

Складу важны BOM, резервы, дефициты, выдача, возврат, сроки и складские комментарии.

Технику важны время, адрес, задачи, документы, схема, контакт ответственного и состав своей зоны работ.

Водителю важны адрес, время, маршрут, транспортная задача, погрузка, общий вес, габариты и детализация груза.

Системному админу проекта как рабочая смета может быть не нужна вообще, если он не участвует в проекте.

## 4. Базовый принцип UI

Карточка проекта должна иметь общий доменный объект `ProjectQuote`, но разные представления:

```text
ProjectQuote
→ Manager View
→ Technical Director View
→ Warehouse View
→ Technician View
→ Driver View
→ Viewer View
→ Admin/System View
```

Это не разные проекты, а разные способы показать один и тот же проект.

## 5. Множественные роли пользователя

### 5.1. Главное решение

Один пользователь может выполнять несколько ролей в одном проекте.

Например:

```text
техник + водитель
менеджер + водитель
технический директор + водитель
техник по звуку + техник по свету
технический директор + техник по конструкциям
```

Это нормальный рабочий сценарий, потому что в реальной работе люди часто заменяют друг друга и могут выполнять две, три или больше ролей.

### 5.2. Как должен работать интерфейс

Интерфейс не должен выбирать только одну роль и скрывать остальные блоки.

Правильная логика:

```text
Пользователь открыл проект
→ система определила все его роли в проекте
→ собрала общий экран из блоков этих ролей
→ скрыла только то, на что нет прав
```

Например, если пользователь одновременно водитель и техник, он должен видеть:

- водительский блок: маршрут, адрес, вес, габариты, груз;
- технический блок: свою зону ответственности, задачи, документы, схему;
- общую шапку проекта: дату, место, время;
- но не видеть финансы, если у него нет финансовых прав.

### 5.3. Роли и права — не одно и то же

Роль описывает рабочую функцию пользователя в проекте.

Право описывает доступ к данным и действиям.

Пример:

```text
Роль: водитель
Роль: техник по звуку
Право: project.view_finance = false
Право: project.view_transport = true
Право: project.view_technical_sections = true
```

Это значит, что пользователь может видеть транспорт и техническую информацию, но не видит общий чек, маржу и внутренние финансовые комментарии.

## 6. Общая верхняя шапка проекта

### 6.1. Обязательное общее правило

Дата, место и время должны быть видны всем пользователям, у которых есть доступ к проекту.

Это не роль-зависимый блок, а общая шапка проекта.

### 6.2. Что должно быть видно всем ролям

Общая шапка проекта должна содержать:

- название или номер проекта;
- дату мероприятия;
- время мероприятия;
- место проведения;
- адрес / площадку, если доступно;
- статус проекта;
- роль пользователя в проекте;
- ближайшее действие пользователя, если оно есть.

### 6.3. Почему дата, место и время обязательны для всех

Любая роль открывает проект в первую очередь для понимания контекста:

```text
Когда?
Где?
Во сколько?
```

Эти данные нужны менеджеру, техдиру, складу, технику, водителю, приглашённому специалисту и пользователю только для просмотра.

## 7. Роль-зависимые блоки

Нужно описать отдельные первые экраны для ролей:

- менеджер / директор;
- технический директор;
- склад;
- техник;
- водитель;
- приглашённый специалист;
- системный администратор;
- viewer / только просмотр.

## 8. Manager / Director View

### 8.1. Назначение

Первый экран менеджера или директора — это коммерческо-организационная сводка проекта.

Он должен быстро отвечать на вопросы:

- кто клиент;
- где мероприятие;
- когда мероприятие;
- кто работает на проекте;
- какая есть субаренда;
- какой общий чек;
- какая маржа.

### 8.2. Что должно быть видно сразу

На первом экране карточки проекта менеджер / директор должен видеть:

- клиента;
- место проведения;
- адрес / площадку;
- дату и время мероприятия;
- состав работников / назначенных пользователей проекта;
- список субаренды;
- общий чек проекта;
- маржу.

### 8.3. Особенности доступа

Маржа и внутренняя финансовая информация должны быть видны только пользователям с соответствующим правом.

Пример права:

```text
project.view_margin
```

Если у менеджера нет права видеть маржу, первый экран должен показывать общий чек без внутренней маржинальности.

### 8.4. Предварительный порядок блоков

```text
1. Общая шапка: дата / место / время / статус
2. Клиент
3. Общий чек
4. Маржа
5. Работники / команда проекта
6. Субаренда
```

Порядок может отличаться между desktop и mobile, но смысл первого экрана должен сохраняться.

## 9. Technical Director View

### 9.1. Назначение

Первый экран технического директора — это расширенная проектно-техническая сводка.

Технический директор должен видеть всё ключевое, что видит менеджер, плюс технические агрегаты проекта.

### 9.2. Что должно быть видно сразу

На первом экране карточки проекта технический директор должен видеть:

- клиента;
- место проведения;
- адрес / площадку;
- дату и время мероприятия;
- состав работников / назначенных пользователей проекта;
- список субаренды;
- общий чек проекта;
- маржу, если есть право на просмотр;
- общий вес проекта;
- рабочую потребляемую мощность;
- пусковую мощность, если она есть или может быть рассчитана.

### 9.3. Технические агрегаты

Технические агрегаты должны собираться из BOM, технических разделов и/или 3D-сцены, если она используется.

Минимальные агрегаты:

```text
project.totalWeightKg
project.totalPowerW
project.totalStartupPowerW
```

Если пусковая мощность неизвестна, интерфейс должен явно показывать, что данных нет, а не подставлять недостоверное значение.

### 9.4. Предварительный порядок блоков

```text
1. Общая шапка: дата / место / время / статус
2. Клиент
3. Технические итоги: вес / мощность / пусковая мощность
4. Общий чек
5. Маржа, если есть право
6. Работники / команда проекта
7. Субаренда
```

Порядок может отличаться между desktop и mobile, но технические агрегаты должны быть видны без глубокого перехода в разделы.

## 10. Warehouse View

### 10.1. Назначение

Первый экран складского пользователя — это складская комплектация проекта.

Склад должен сразу понимать:

- какой проект открыт;
- где и когда мероприятие;
- какие позиции нужно подготовить со своего склада;
- какие позиции не идут со склада и закрываются субарендой;
- кто является поставщиком субаренды;
- какие количества требуются.

### 10.2. Что должно быть видно сразу

На первом экране карточки проекта складской пользователь должен видеть:

- название / номер проекта;
- дату мероприятия;
- место проведения;
- время мероприятия;
- дату и время ближайшего складского действия, если оно отличается от времени мероприятия;
- список всех наименований, которые идут со склада;
- количество по каждой складской позиции;
- отдельный список позиций в субаренду;
- поставщика / владельца субаренды по каждой субарендной позиции;
- статус подготовки / резерва, если он уже есть.

### 10.3. Разделение списков

Для склада важно не смешивать собственный склад и субаренду в одну кашу.

Первый экран должен иметь минимум два отдельных блока:

```text
1. Со склада
2. Субаренда
```

В блоке `Со склада` показываются позиции, которые нужно собрать из собственного склада.

В блоке `Субаренда` показываются позиции, которые закрываются внешним поставщиком, с указанием `от кого`.

### 10.4. Предварительный порядок блоков

```text
1. Общая шапка: проект / дата / место / время / статус
2. Ближайшее складское действие
3. Со склада: наименование + количество + статус
4. Субаренда: наименование + количество + поставщик
5. Общий статус складской подготовки
```

Финансовые данные, общий чек и маржа складскому пользователю на первом экране не нужны, если только ему явно не выданы дополнительные права.

## 11. Technician / Executor View

### 11.1. Главное решение

Техников стоит разделять по направлениям, но не изолировать жёстко.

Правильная модель:

```text
У техника есть направление / роль / зона ответственности
→ его направление показывается первым
→ остальные нефинансовые данные проекта доступны глубже
→ финансовая информация скрыта
```

Техник по звуку первым видит звук, техник по свету — свет, техник по конструкциям — сцену, фермы, LED или конструктив. Но если ему нужно открыть общую схему, адрес, состав проекта, документы или другие технические разделы, он не должен упираться в пустой экран.

### 11.2. Почему не стоит жёстко резать техников по направлениям

На реальном мероприятии направления связаны между собой:

- звук зависит от сцены, питания, коммутации и схемы площадки;
- свет зависит от ферм, высот, питания и схемы;
- LED зависит от конструкций, питания, подвеса и места установки;
- конструкции зависят от сцены, площадки, веса, схемы и доступа;
- техник может помогать не только в своей зоне.

Если скрывать от техника всё, кроме его направления, это может мешать работе на площадке.

### 11.3. Что должно быть видно сразу

На первом экране карточки проекта техник / исполнитель должен видеть:

- общую шапку: дату, место, время, статус;
- свою роль на проекте;
- свою зону ответственности;
- ближайшее действие или задачу;
- список задач, назначенных именно ему;
- технические разделы, относящиеся к его направлению;
- быстрый доступ к общей схеме / 3D / PDF, если они есть;
- контакты ответственного менеджера или технического директора;
- важные комментарии по площадке, монтажу и демонтажу.

### 11.4. Группировка технической информации

Техническую информацию нужно группировать по категориям:

```text
Звук
Свет
Сцена
Фермы
LED
Backline
Коммутация
Питание
Транспорт / погрузка
Общие документы
```

Для техника его категория должна быть раскрыта первой или подсвечена.

Остальные категории должны быть доступны, но не должны забивать первый экран.

### 11.5. Что скрывается от техника

По умолчанию техник / приглашённый специалист не должен видеть:

- общий чек проекта;
- маржу;
- внутреннюю себестоимость;
- ставки других исполнителей;
- коммерческие условия клиента;
- внутренние финансовые комментарии.

Эти данные могут быть доступны только при отдельном праве.

### 11.6. Предварительный порядок блоков

```text
1. Общая шапка: дата / место / время / статус
2. Моя роль и моя зона ответственности
3. Мои ближайшие задачи
4. Мой технический раздел
5. Схема / документы / чеклист
6. Контакты ответственных
7. Остальные технические разделы, свернутые по категориям
```

## 12. Driver View

### 12.1. Назначение

Первый экран водителя — это транспортная карточка проекта.

Водитель должен сразу понимать:

- куда ехать;
- когда ехать;
- что везти по массе и габаритам;
- какой маршрут;
- где адрес и точка назначения.

### 12.2. Что должно быть видно сразу

На первом экране карточки проекта водитель должен видеть:

- общую шапку: дату, место, время, статус;
- адрес площадки;
- карту маршрута до площадки;
- точку назначения;
- общий вес груза / проекта;
- габариты груза, если они есть;
- комментарий менеджера с подробностями маршрута, заезда, парковки, разгрузки или доступа на площадку;
- контакт ответственного по проекту или площадке.

### 12.3. Маршрут на старте

На первых этапах достаточно одного адреса площадки и маршрута до площадки.

Минимальный сценарий:

```text
точка отправления / склад → адрес площадки
```

Если точка отправления неизвестна или не ведётся в системе, можно показывать только адрес площадки и открытие маршрута во внешней карте.

Менеджер должен иметь возможность оставить комментарий с подробностями:

- куда подъезжать;
- где ворота;
- где парковка;
- как связаться на площадке;
- особенности разгрузки;
- ограничения по высоте, весу, времени или пропускам.

### 12.4. Фундамент под сложные маршруты

В первой версии не нужно делать сложный маршрутный планировщик.

Но модель транспорта должна позволять позже добавить несколько точек маршрута:

```text
склад → субарендодатель → площадка → склад
```

или:

```text
склад → площадка A → площадка B → склад
```

Для этого в модели можно заложить поле:

```text
project.transport.routePoints
```

На старте оно может содержать одну точку назначения — адрес площадки.

### 12.5. Детализация груза

Водитель должен иметь возможность провалиться в карточку груза и увидеть детализацию по позициям.

Это особенно важно, потому что водителем может быть техник, менеджер или другой участник проекта, который параллельно выполняет несколько функций.

Детализация груза должна включать:

- список позиций;
- количество;
- вес, если известен;
- габариты, если известны;
- принадлежность к разделу проекта;
- пометку `со склада` или `субаренда`;
- комментарии по погрузке / разгрузке;
- особые условия перевозки, если есть.

На первом экране можно показывать только общий вес и габариты, но внутри транспортного блока должен быть доступен полный список груза.

### 12.6. Транспортные агрегаты

Минимальные поля для транспортного представления:

```text
project.transport.totalWeightKg
project.transport.dimensions
project.transport.route
project.transport.routePoints
project.transport.destinationAddress
project.transport.managerRouteComment
project.transport.accessNotes
project.transport.cargoItems
```

Если габариты неизвестны, интерфейс должен показывать `габариты не указаны`, а не подставлять случайные значения.

### 12.7. Предварительный порядок блоков

```text
1. Общая шапка: дата / место / время / статус
2. Адрес и карта маршрута до площадки
3. Комментарий менеджера по маршруту / заезду / разгрузке
4. Общий вес
5. Габариты, если есть
6. Кнопка / блок детализации груза
7. Контакты ответственных
```

Финансовые данные, общий чек и маржа водителю на первом экране не нужны, если только ему явно не выданы дополнительные права.

### 12.8. Статусы водителя

На старте водитель не должен вести трекинг статусов по задаче.

Не нужны обязательные кнопки:

```text
выехал
прибыл на склад
загрузился
прибыл на площадку
разгрузился
вернулся
```

Водительский экран первой версии должен быть информационным, а не диспетчерским.

## 13. Открытые вопросы

### Open Question CARD.1 — Общая шапка

Закрыто первым решением: дата, место и время должны быть видны всем пользователям, у которых есть доступ к проекту.

Остаётся уточнить:

- всегда ли всем виден клиент;
- всегда ли всем виден полный адрес;
- нужно ли скрывать часть данных для приглашённых специалистов;
- должен ли статус проекта быть виден всем.

### Open Question CARD.2 — Менеджер / директор

Закрыто первым решением: клиент, место, время, состав работников, список субаренды, общий чек и маржа.

Остаётся уточнить:

- нужна ли на первом экране кнопка быстрого экспорта PDF;
- нужен ли статус складской проверки;
- нужен ли статус согласования с клиентом;
- нужна ли история последних изменений.

### Open Question CARD.3 — Технический директор

Закрыто первым решением: технический директор видит всё то же, что менеджер/директор, плюс вес, потребляемую мощность и пусковую мощность при наличии данных.

Остаётся уточнить:

- нужны ли на первом экране технические предупреждения и риски;
- нужен ли быстрый доступ к чеклисту площадки;
- нужен ли быстрый доступ к 3D-схеме;
- нужна ли сводка по монтажу и демонтажу.

### Open Question CARD.4 — Склад

Закрыто первым решением: складской пользователь видит дату, место, время, список проекта, все наименования, которые идут со склада, и отдельный список субаренды с указанием поставщика.

Остаётся уточнить:

- должен ли склад сразу видеть чекбоксы сборки по позициям;
- нужен ли режим сканирования / штрихкодов;
- должен ли склад видеть дефициты отдельным третьим блоком;
- нужны ли комментарии склада на первом экране.

### Open Question CARD.5 — Техник / исполнитель

Закрыто первым решением: техников нужно разделять по направлениям, но не блокировать от общей нефинансовой информации проекта. Их направление показывается первым, остальные технические категории доступны глубже. Финансовая информация скрыта.

Остаётся уточнить:

- какие направления техников нужны в первой версии;
- нужно ли разрешить одному технику иметь несколько направлений;
- какие действия техник может отмечать выполненными;
- должен ли приглашённый специалист отличаться от штатного техника по видимости данных.

### Open Question CARD.6 — Водитель

Закрыто первым решением: водитель видит дату, место, время, адрес, карту маршрута до площадки, комментарий менеджера, общий вес, габариты при наличии данных и детализацию груза при провале в транспортный блок.

Также закрыто: водитель не отмечает статусы `выехал`, `прибыл`, `разгрузился`, `вернулся` в первой версии.

На будущее заложен фундамент под несколько точек маршрута через `project.transport.routePoints`, но в первой версии достаточно одного адреса площадки и маршрута до неё.

Остаётся уточнить:

- нужно ли показывать время погрузки и разгрузки отдельно от времени мероприятия;
- как объединять водительский экран с другими ролями пользователя.

Должен ли системный администратор видеть проектные данные по умолчанию или только при наличии проектных прав?
