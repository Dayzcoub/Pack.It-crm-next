# 00. Продуктовые принципы PACK.IT

## 1. PACK.IT — это проектно-сметная CRM

Главный продуктовый принцип: **PACK.IT строится вокруг проекта/сметы**.

Приложение должно позволять пользователю быстро создать проект, собрать технический пакет, посчитать стоимость, проверить склад, оформить документы и довести проект до нужного статуса.

3D-конструктор является важным профессиональным инструментом, но не является обязательным входом в приложение.

## 2. Главная сущность — Проект / Смета

Проект объединяет:

- клиента;
- площадку;
- даты мероприятия, монтажа и демонтажа;
- технические разделы;
- коммерческую смету;
- техническую комплектацию;
- складские резервы;
- дефициты и субаренду;
- команду;
- транспорт;
- документы;
- статусы;
- историю изменений.

## 3. 3D-конструктор подключается по необходимости

В проекте должен быть отдельный чекпоинт:

```text
Нужна техническая 3D-сцена?
Да  → открыть 3D-конструктор
Нет → продолжить обычную смету
```

Проект не должен зависеть от наличия 3D-сцены.

Примеры проектов без 3D:

- аренда звукового комплекта;
- работа технической команды;
- доставка оборудования;
- простой расчёт услуг;
- небольшой корпоратив;
- отдельная аренда света, экрана или коммутации.

Примеры проектов с 3D:

- сцена с настилами;
- ферменный портал или рама;
- LED-экран;
- техническая схема площадки;
- расстановка оборудования;
- визуализация для клиента или технической команды.

## 4. Склад подключается к проекту

Склад не является центром приложения.

Склад обслуживает проект:

```text
Проект запрашивает оборудование
→ склад проверяет наличие
→ создаётся резерв, дефицит или субаренда
```

Проект должен уметь жить на разных стадиях:

- черновик без складской проверки;
- предварительная проверка наличия;
- резерв;
- подтверждение;
- сборка;
- выдача;
- возврат;
- закрытие.

## 5. Системная админка отделена от склада и проектов

Администратор приложения — это не складской пользователь и не менеджер проекта.

Системная админка отвечает за само приложение:

- пользователей;
- роли;
- доступы;
- ключи и приглашения;
- настройки компании;
- логотипы;
- фирменный стиль;
- PDF-шаблоны;
- системные справочники;
- доступные модули;
- аудит действий;
- общие настройки программы.

## 6. Старое приложение — донор логики, но не архитектуры

Старое приложение используется только как источник:

- проверенных формул;
- расчётной логики;
- бизнес-сценариев;
- справочников;
- правил формирования комплектации;
- идей для интерфейса, которые уже показали пользу.

Старый код, старую структуру подключения файлов, старые UI-костыли и случайную архитектуру нельзя переносить как основу нового продукта.

## 7. Mobile-first не означает урезанный продукт

Мобильная версия не должна быть уменьшенной копией десктопа.

На мобильном устройстве пользователь должен решать реальные задачи:

- быстро открыть проект;
- посмотреть состав сметы;
- проверить статус;
- внести правку;
- посмотреть складскую доступность;
- открыть PDF;
- принять решение;
- отправить документ;
- отметить действие на площадке.

Сложные действия должны быть разбиты на пошаговые сценарии.

## 8. Документы — часть продукта

PDF, КП, техническое задание, складской лист, монтажный лист и внутренняя техническая сводка должны считаться не побочным экспортом, а полноценным модулем продукта.

Пользователь должен иметь возможность настраивать:

- логотип компании;
- реквизиты;
- внешний вид PDF;
- структуру коммерческого предложения;
- видимость технических данных;
- шаблоны документов;
- язык и формулировки документов.
