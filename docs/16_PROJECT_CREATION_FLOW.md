# 16. Создание проекта

## 1. Статус документа

Этот документ фиксирует стартовый сценарий создания проекта PACK.IT.

## 2. Главное решение

На первом шаге создания проекта обязательны только базовые данные, без которых проект теряет смысл как рабочая сущность.

Дата мероприятия обязательна, потому что без неё невозможно проверить занятость оборудования, складские резервы и доступность ресурсов.

Название проекта вводится вручную, потому что это человеческое рабочее имя проекта.

Номер проекта генерируется автоматически системой и использует короткий префикс компании, которая работает в PACK.IT.

При этом время старта мероприятия желательно указать сразу, но оно не должно блокировать создание проекта.

## 3. Обязательные поля первого шага

При создании нового проекта пользователь заполняет:

- название проекта;
- клиент;
- место проведения;
- адрес;
- дата мероприятия.

Система автоматически создаёт:

- номер проекта.

Желательное, но не обязательное поле:

- время старта мероприятия.

## 4. Почему именно эти поля важны

Эти данные формируют минимальный контекст проекта:

```text
Что это за проект?
Кому делаем?
Где делаем?
Куда ехать?
На какую дату проверять доступность?
Какой системный номер у проекта и документов?
```

Название проекта нужно для быстрого человеческого поиска и понимания сути события.

Номер проекта нужен как техническая сквозная привязка документов.

Клиент, место, адрес и дата нужны для создания рабочей карточки проекта.

Дата обязательна, потому что по ней система должна показывать:

- какое оборудование уже занято;
- какое оборудование свободно;
- какие складские резервы пересекаются;
- есть ли потенциальные конфликты по ресурсам.

Время старта мероприятия желательно, потому что оно помогает календарю, команде, складу и транспорту. Но на раннем этапе клиент может ещё не знать точное время, поэтому отсутствие времени не должно блокировать создание проекта.

## 5. Название проекта

Название проекта пользователь должен вводить вручную.

Это не технический идентификатор и не автоматическая склейка клиента, места и даты.

Примеры нормальных названий:

```text
Концерт группы Арай
Свадьба Ивановых
Юбилей 50 лет
Корпоратив в шатре
Городской праздник
Презентация продукта
```

Система может в будущем предлагать автоподсказку, но основное решение — ручное название.

## 6. Номер проекта

Номер проекта должен генерироваться автоматически.

Это системный идентификатор, который должен использоваться для связи проекта со всеми документами и внутренними сущностями.

Префикс номера должен быть привязан к названию или настройкам компании, которая пользуется программой.

PACK.IT не должен навязывать свой бренд в номерах рабочих документов компании.

Принцип номера:

```text
Префикс компании + год + порядковый номер
```

Номер проекта должен привязываться к:

- номеру сметы;
- коммерческому предложению / КП;
- складским документам;
- техническим документам;
- PDF-файлам;
- резервам;
- задачам;
- календарным событиям;
- истории проекта.

Формат номера и правила генерации нужно будет настроить в системной админке компании.

Главное правило: пользователь не должен вручную придумывать номер проекта на старте.

## 7. Префикс компании

Префикс компании должен быть коротким и понятным.

Если компания называется длинно, система не должна использовать всё название в номере проекта.

Правило первой версии:

```text
companyPrefix = 3–5 букв
```

Система должна предложить префикс автоматически из названия компании, но админ должен иметь возможность изменить его вручную.

Пример: если компания называется `Свежее решение`, система может предложить короткий префикс из 3–5 букв, собранный из значимых слов названия.

Требования к префиксу:

- короткий;
- читаемый;
- понятный владельцу компании;
- без длинных фраз;
- без пробелов;
- лучше латиницей для документов и файлов;
- уникальный внутри системы или рабочего пространства.

Префикс хранится в настройках компании / рабочего пространства и используется при генерации новых номеров проектов.

## 8. Клиент

Клиент должен выбираться из базы клиентов или создаваться прямо из формы проекта.

Минимальная логика:

```text
Выбрать существующего клиента
или
Создать нового клиента
```

## 9. Место и адрес

`Место` и `адрес` лучше не смешивать в одно поле.

Пример:

```text
Место: ДК / ресторан / шатёр / частная площадка / офис / улица / другое
Адрес: конкретный адрес для маршрута, документов, склада и команды
```

Место может быть названием площадки, а адрес — географической точкой.

## 10. Дата мероприятия

Дата мероприятия обязательна на первом шаге создания проекта.

Правило первой версии:

```text
eventDate = обязательно
```

Дата используется не только для календаря, но и для проверки доступности оборудования.

После выбора даты система должна иметь возможность показать, что на эту дату:

```text
занято
свободно
в резерве
в конфликте
```

## 11. Время мероприятия

На первом шаге время означает старт мероприятия.

Это не интервал `с / по` и не расписание монтажа.

Правило первой версии:

```text
eventStartAt = желательно, но не обязательно
```

Если время не указано, проект всё равно можно создать, но интерфейс должен показывать, что время мероприятия ещё не заполнено.

## 12. Статусы проекта

### 12.1. Главное решение

Статусы `Подтверждён` и `Предоплачен` не нужно разделять на два разных рабочих статуса.

Их нужно слить в один статус:

```text
confirmed
```

Русское отображение:

```text
Подтверждён / предоплачен
```

Смысл статуса: проект уже достаточно реален, чтобы готовить склад, логистику, монтаж, демонтаж, команду и технические задачи.

### 12.2. Базовый набор статусов

Для старта подходит такой набор:

```text
draft        → Черновик
in_progress  → В работе
sent         → Отправлен клиенту
confirmed    → Подтверждён / предоплачен
completed    → Завершён
cancelled    → Отменён
```

Отдельный статус `prepaid` на старте не нужен.

## 13. Время склада, монтажа и демонтажа

Время загрузки на складе, монтажа и демонтажа можно указывать уже после того, как проект переведён в статус:

```text
confirmed
```

Рабочее русское отображение:

```text
Подтверждён / предоплачен
```

После этого проект уже достаточно реален, чтобы планировать:

- загрузку на складе;
- выезд;
- монтаж;
- демонтаж;
- возврат;
- складскую подготовку;
- транспорт.

## 14. Что не обязательно на первом шаге

На первом шаге не нужно требовать сразу:

- ручной номер проекта;
- точное время старта мероприятия;
- время загрузки на складе;
- время монтажа;
- время демонтажа;
- полный состав сметы;
- складскую комплектацию;
- транспорт;
- команду;
- 3D-сцену;
- PDF;
- субаренду;
- финальную цену;
- маржу;
- чеклист площадки.

Эти данные можно добавлять после создания проекта.

## 15. Предварительная модель

```text
ProjectCreateInput
├── title
├── clientId | newClientDraft
├── venueName
├── venueAddress
├── eventDate
├── eventStartAt?
└── optionalComment
```

Системные поля после создания:

```text
Project
├── id
├── projectNumber
├── companyPrefix
├── title
├── clientId
├── venueName
├── venueAddress
├── eventDate
├── eventStartAt?
└── status
```

`title` является обязательным и вводится вручную.

`projectNumber` генерируется автоматически.

`companyPrefix` берётся из настроек компании / рабочего пространства и должен быть коротким, 3–5 букв.

`eventDate` является обязательным.

`eventStartAt` является опциональным.

`eventEndAt` не нужен на первом шаге создания проекта, если точный интервал ещё неизвестен.

## 16. Статусная логика расписания

Пока проект находится в раннем статусе, например:

```text
draft
in_progress
sent
```

достаточно хранить базовую информацию проекта и проверять доступность оборудования по дате проекта.

Когда проект переходит в:

```text
confirmed
```

появляется смысл заполнять календарные и логистические события:

```text
warehouse_load
setup
teardown
transport
warehouse_return
```

## 17. Открытые вопросы

### Open Question CREATE.1 — Дата

Закрыто первым решением: дата мероприятия обязательна на первом шаге, потому что нужна для проверки занятости и доступности оборудования.

### Open Question CREATE.2 — Статусы

Закрыто первым решением: статусы `Подтверждён` и `Предоплачен` объединяются в один статус `confirmed` с отображением `Подтверждён / предоплачен`.

### Open Question CREATE.3 — Название проекта

Закрыто первым решением: название проекта вводится вручную. Это рабочее человеческое имя события, например `Концерт группы Арай`, `Свадьба`, `Юбилей`.

### Open Question CREATE.4 — Номер проекта

Закрыто первым решением: номер проекта генерируется автоматически, использует префикс компании и применяется как сквозная привязка для сметы, КП, складских документов, технических документов и истории проекта.

### Open Question CREATE.5 — Префикс компании

Закрыто первым решением: префикс компании генерируется коротким, 3–5 букв, из названия компании. Админ может изменить предложенный префикс вручную.

### Open Question CREATE.6 — Формат номера проекта

Как именно формировать номер проекта из префикса компании, года и порядкового номера?

### Open Question CREATE.7 — Адрес

Адрес должен быть свободным текстом или нужна структура с геоточкой / картой?
