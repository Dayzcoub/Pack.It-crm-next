# 02. Доменная модель Проекта / Сметы

## 1. Главная сущность

Главная рабочая сущность PACK.IT — **Проект / Смета**.

В интерфейсе пользователь может воспринимать её как смету, проект, заказ или мероприятие, но в архитектуре это один центральный объект.

Рабочее название доменной сущности:

```text
ProjectQuote
```

## 2. Задача ProjectQuote

ProjectQuote должен объединять:

- коммерческую часть;
- техническую часть;
- клиента;
- площадку;
- даты;
- команду;
- транспорт;
- складскую потребность;
- документы;
- статусы;
- историю;
- опциональную 3D-сцену.

## 3. Базовая структура ProjectQuote

```text
ProjectQuote
├── identity
├── client
├── venue
├── schedule
├── status
├── sections
├── optionalScene
├── bom
├── inventoryState
├── crew
├── transport
├── documents
├── totals
├── permissions
└── history
```

## 4. Identity

Идентификация проекта:

- id;
- номер проекта;
- название;
- рабочее описание;
- workspace/company id;
- автор;
- ответственный менеджер;
- дата создания;
- дата обновления.

## 5. Client

Клиент:

- тип клиента: физическое лицо / компания;
- имя или название;
- контактное лицо;
- телефон;
- email;
- реквизиты;
- комментарий;
- история проектов клиента.

## 6. Venue

Площадка:

- название;
- адрес;
- контакт площадки;
- особенности заезда;
- лифт / пандус / этаж;
- электричество;
- ограничения по времени;
- комментарии;
- файлы и фото площадки в будущем.

## 7. Schedule

Даты:

- дата мероприятия;
- период мероприятия;
- дата монтажа;
- время монтажа;
- дата демонтажа;
- время демонтажа;
- смены;
- дедлайны подготовки;
- внутренние контрольные точки.

## 8. Status

Проект должен иметь управляемый статус.

Предварительный набор:

```text
draft
in_progress
waiting_client
approved
reserved
in_preparation
on_site
completed
cancelled
archived
```

Статусы должны быть расширяемыми через настройки.

## 9. Sections

Разделы — это содержательная часть сметы.

Примеры разделов:

- Stage;
- Truss;
- LED;
- Sound;
- Light;
- Backline;
- Commutation;
- Services;
- Crew;
- Transport;
- Custom.

Раздел может быть:

- ручным;
- собранным из справочника;
- рассчитанным формулой;
- созданным из 3D-сцены;
- импортированным;
- помеченным как субаренда.

## 10. Optional Scene

3D-сцена является опциональной частью проекта.

```text
optionalScene: null | SceneModel
```

Если 3D не нужен, поле может отсутствовать или быть пустым.

Если 3D используется, сцена должна быть связана с проектными разделами и BOM.

## 11. BOM

BOM — техническая комплектация проекта.

BOM может формироваться из:

- ручных строк сметы;
- технических разделов;
- 3D-сцены;
- шаблонов;
- складских комплектов;
- импортированных данных.

Строка BOM должна содержать минимум:

- id;
- sectionId;
- sourceType;
- itemId;
- name;
- code;
- quantity;
- unit;
- weight;
- power;
- price;
- inventoryItemId;
- availability;
- deficitQty;
- subrentQty;
- notes.

## 12. Inventory State

Складское состояние проекта:

- склад ещё не проверен;
- есть в наличии;
- частичный дефицит;
- требуется субаренда;
- зарезервировано;
- выдано;
- возвращено;
- закрыто.

Складское состояние не должно ломать коммерческую смету. Пользователь может сначала собрать коммерческое предложение, а склад проверить позже.

## 13. Crew

Команда проекта:

- роль;
- пользователь;
- приглашённый специалист;
- ставка;
- часы;
- смены;
- комментарии;
- доступ к проекту.

В клиентских документах может показываться только роль, без имени конкретного исполнителя.

Во внутренних документах может показываться роль и имя.

## 14. Transport

Транспорт:

- тип транспорта;
- количество машин;
- маршрут;
- стоимость;
- водитель;
- комментарий;
- связь со складской выдачей.

## 15. Documents

Документы проекта:

- коммерческое предложение;
- техническая сводка;
- складской лист;
- монтажный лист;
- внутренний расчёт;
- схема сцены;
- акт / закрывающие документы в будущем.

Документы должны генерироваться из ProjectQuote, а не из случайного состояния интерфейса.

## 16. Totals

Итоги:

- сумма аренды;
- сумма услуг;
- сумма транспорта;
- сумма команды;
- сумма субаренды;
- скидка;
- маржа;
- налоговые поля в будущем;
- общий итог для клиента;
- внутренний итог.

## 17. Permissions

Права внутри проекта:

- кто может смотреть проект;
- кто может редактировать коммерческую часть;
- кто может редактировать техническую часть;
- кто может менять складские резервы;
- кто может утверждать проект;
- кто может экспортировать документы;
- кто может видеть внутреннюю себестоимость.

## 18. History

История проекта должна фиксировать важные действия:

- создание;
- изменение статуса;
- изменение клиента;
- изменение суммы;
- изменение технических разделов;
- создание резерва;
- изменение PDF-настроек;
- назначение команды;
- экспорт документов;
- выдача со склада;
- возврат;
- закрытие.

## 19. Главный принцип модели

ProjectQuote не должен быть просто набором полей интерфейса.

Это доменная модель, из которой должны строиться:

- экран проекта;
- смета;
- BOM;
- складская потребность;
- PDF;
- история;
- отчёты;
- статусы;
- интеграции.
