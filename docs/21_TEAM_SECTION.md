# 21. Команда

## Решение

`Команда` — отдельный раздел сметы и проекта.

Команда не является частью раздела `Услуги`, потому что это не абстрактные работы, а конкретные роли и назначенные люди внутри проекта.

## Клиентский вывод

В клиентской смете команда показывается отдельным разделом.

Клиент видит:

- роль;
- количество;
- стоимость.

Клиент не видит имена конкретных людей.

Пример клиентского вывода:

```text
Команда
Звукорежиссёр — 1 — цена Y
Техник сцены — 2 — цена Y
Грузчики — 4 — цена Y
```

Формула расчёта клиенту по умолчанию не показывается.

## Внутреннее использование

Внутри проекта система должна хранить конкретных назначенных людей.

Внутри проекта видны:

- роль;
- назначенный человек;
- контакт / профиль пользователя, если есть;
- ставка;
- тип расчёта;
- часы / смены;
- внутренняя себестоимость;
- цена клиенту;
- комментарий.

Назначенные люди нужны для:

- календаря;
- задач;
- технических листов;
- доступа к проекту;
- расчёта внутренней себестоимости;
- контроля маржи;
- коммуникации внутри команды.

## Стартовые роли

На старте в справочник ролей команды входят:

- технический директор;
- звукорежиссёр;
- светорежиссёр;
- видеоинженер / LED-инженер;
- техник сцены;
- техник ферм;
- техник LED;
- монтажник;
- грузчик;
- водитель;
- электрик;
- менеджер проекта;
- администратор площадки;
- приглашённый специалист;
- другое.

Этот список является стартовым. Если в тестировании или реальной работе появятся новые повторяющиеся роли, справочник можно расширить без переделки общей логики раздела `Команда`.

## Приглашённые специалисты и временные ключи

Для приглашённых специалистов в CRM Next нужно заложить логику временных ключей доступа для зарегистрированного пользователя.

В донорском проекте уже есть похожая логика временных ключей. Её можно использовать как источник бизнес-правил и сценариев, но в CRM Next реализацию нужно адаптировать под новую архитектуру доступа и проектные роли.

Базовое правило:

- приглашённый специалист должен быть зарегистрированным пользователем;
- ему можно выдать временный ключ / временный доступ;
- доступ привязывается к конкретному проекту;
- доступ имеет дату начала и окончания;
- доступ ограничивается назначенной ролью;
- пользователь видит только разрешённые ему задачи, документы и проектные данные;
- после окончания срока временный доступ закрывается.

Такой доступ нужен для внешних специалистов, субподрядчиков и временных участников команды, которых не нужно постоянно держать в полном рабочем пространстве.

## Типы расчёта

На старте поддерживаются два типа расчёта:

- за смену / мероприятие;
- почасовой расчёт.

Клиенту показывается итоговая стоимость строки, а не внутренняя формула.

## Ставки, клиентская цена и маржа

Строка команды должна быть гибкой сметной позицией.

Для каждой строки команды хранятся:

- роль;
- количество;
- тип расчёта: смена / час / фикс;
- количество смен или часов;
- внутренняя ставка;
- внутренняя сумма / выплата;
- цена клиенту;
- клиентская сумма;
- расчётная маржа;
- назначенный человек;
- комментарий;
- показывать клиенту: да / нет;
- группировать в клиентской смете: да / нет.

Формула маржи:

```text
маржа = клиентская сумма − внутренняя сумма
```

Система не должна навязывать автоматическую наценку. Она может подставлять базовые ставки из справочника, но менеджер может изменить внутреннюю ставку и цену клиенту в конкретном проекте.

Назначенный человек должен видеть сумму, которую он получит за работу на этой площадке.

Назначенный человек не должен видеть клиентскую цену, маржу проекта и чужие выплаты, если его роль не даёт такого доступа.

Клиент видит только клиентскую стоимость. По решению менеджера команда может выводиться детально по ролям или сворачиваться в общий блок.

## Видимость команды в клиентской смете

Для клиентской сметы у раздела `Команда` есть три режима видимости:

- `детально` — роли выводятся отдельными строками;
- `общим блоком` — одна строка, например `Техническая команда`;
- `скрыть из клиентской сметы` — команда остаётся только во внутренней себестоимости проекта.

В любом режиме внутри проекта сохраняется полная детализация по ролям, ставкам, назначенным людям, выплатам и марже.

## Статусы участия и ознакомления

Для назначенных людей нужны статусы участия и ознакомления с задачей / проектом.

Стартовые статусы:

- `назначен`;
- `ожидает подтверждения`;
- `подтвердил`;
- `отказался`;
- `замена требуется`;
- `закрыл работу`.

Статусы нужны, чтобы менеджер, технический директор и руководство понимали:

- кто уже назначен;
- кто увидел назначение;
- кто изучил задачу и вводные по площадке;
- кто подтвердил участие;
- кто отказался;
- где нужна замена;
- кто закрыл свою работу после мероприятия.

Подтверждение участия должно фиксировать факт ознакомления исполнителя с задачей / проектом, а не быть только формальной отметкой.

По возможности система должна хранить время ключевых действий: когда назначили, когда исполнитель открыл задачу, когда подтвердил, когда отказался, когда закрыл работу.

## Разграничение с услугами

`Услуги` — это работы и операции, например монтаж, настройка, демонтаж, погрузка, доставка как услуга.

`Команда` — это роли и люди, которые выполняют работу в проекте.

Эти разделы не нужно смешивать в клиентской смете и во внутренней структуре проекта.
